# mix_skill_people

## 30 秒版本

這個專案想證明一件事：

**我可以根據客戶的需求，客製出他們要的 AI 專家、AI 人格，以及 AI 的做事方式。**

不是只有寫幾句 prompt。

而是把客戶要的：

- 專業判斷
- 問答方式
- 溝通語氣
- 工作流程
- 風險邊界
- 決策標準

整理成一套 AI 真的能穩定執行的 `agent skill`。

這個 repo，就是這件事的證明。

---

## 如果你是客戶，你最需要知道的是什麼

如果你要的不是一個普通聊天機器人，而是：

- 像顧問一樣會追問、會分析的 AI
- 像產品經理一樣會拆需求的 AI
- 像技術主管一樣會 review 方案的 AI
- 像客服主管一樣有固定話術與標準的 AI
- 像公司內部專家一樣懂你們 SOP、規則和風格的 AI

那我做的事情，就是把這些需求轉成 AI 可以真的照做的 skill system。

你可以把它理解成：

**我不是只做 AI 功能，我是在做 AI 的「專業角色設計」與「人格工程」。**

---

## 這個專案到底在做什麼

這個專案不是在做單一 prompt，也不是在堆一堆指令。

它做的是更底層、也更有價值的事情：

**先把成熟的 AI skill system 拆開研究，理解它們怎麼讓 AI 變成不同類型的專家，再把這些能力重新組裝成客戶真正需要的版本。**

這次我選了兩套已經很成熟的 skill repo 來做拆解：

- `gstack`
- `superpowers`

我做的不是單純下載，而是：

1. 完整讀取裡面的 skill 文件
2. 抽出每一份 skill 真正的核心
3. 把它們整理成可比較、可分類、可重組的資料
4. 再進一步推導出可以怎麼合成新的 AI 專家人格

所以這個 repo 的價值，不只是內容本身，而是它展示了一個能力：

**我有能力把複雜的 skill 文件，拆解成可交付的 AI 行為系統。**

---

## 為什麼客戶會需要這件事

很多公司以為導入 AI，只要：

- 接上模型
- 寫一段 system prompt
- 丟一些文件進去

就夠了。

但真正上線之後，問題通常會變成：

- AI 今天像顧問，明天像客服，後天像亂講話的助理
- AI 明明讀過文件，但回話還是很泛
- AI 不會追問、不會判斷、不知道什麼時候該保守
- AI 的語氣不符合品牌
- AI 沒有固定標準，回答品質不穩
- AI 知道很多資訊，但不知道怎麼照公司流程做事

也就是說，問題不在模型不夠大，而在：

**AI 沒有被塑造成一個真正有角色、有判斷、有邊界的人。**

我做的事情，就是把這件事補起來。

## 我能幫客戶做出哪些類型的 agent

### 1. 專家型 agent

把 AI 做成某一類專家，而不是泛用助理。

例如：

- 產品策略顧問
- 技術架構 reviewer
- code review 專家
- 合規 / 風控助理
- 客服品質審核員
- 銷售陪跑顧問
- 教學教練型 agent

### 2. 人格型 agent

讓 AI 不只是能回答，而是有穩定的個性與互動方式。

例如：

- 嚴謹型：會追問、會挑戰假設、不輕易放過漏洞
- 陪跑型：白話、鼓勵、擅長引導與拆解問題
- 主管型：直接、判斷清楚、重視結果與風險
- 品牌型：語氣固定、符合品牌風格、說話一致

### 3. 流程型 agent

讓 AI 不只是自由發揮，而是照固定流程做事。

例如：

- 客服回覆流程
- 需求澄清流程
- 專案 kickoff 流程
- 審查流程
- 上線前驗收流程
- 內部知識問答流程

### 4. 混合型 agent

這通常是最有價值的版本。

也就是把：

- 專家知識
- 人格風格
- 提問策略
- 決策邏輯
- 工作流程
- 禁止事項

全部組在一起，變成一個可以真的工作的 AI。

這時候客戶拿到的就不是一個「很會聊天的模型」，而是一個：

- 知道自己是誰
- 知道現在在做什麼
- 知道怎麼判斷
- 知道怎麼跟人說話
- 知道什麼該做、什麼不能做

的 agent。

## 我是怎麼把客戶需求變成 agent skill 的

這件事不能只靠靈感。

我的做法是把需求拆成幾個很清楚的層：

### 1. 角色

先定義 AI 到底是誰。

它是：

- 顧問
- PM
- reviewer
- 客服
- 內部專家
- 培訓教練

不同角色，問法、判斷方式、風險取向都會不同。

### 2. 人格

再定義它應該怎麼說話、怎麼互動。

例如：

- 要不要強勢
- 要不要保守
- 要不要會追問
- 要不要白話
- 要不要先安撫再糾正
- 要不要像主管一樣直接

### 3. 思考框架

不是只讓 AI 知道答案，而是讓它知道怎麼想。

例如：

- 是先問清楚再答
- 還是先給建議再補條件
- 是先找風險
- 還是先找機會
- 是偏重完整性
- 還是偏重速度

### 4. 工作流程

定義這個 agent 在某個場景下，應該照什麼順序做事。

例如：

1. 先確認需求
2. 再找缺口
3. 再做判斷
4. 最後輸出建議

### 5. 規則與邊界

這裡最重要。

因為很多 AI 失控，不是因為不夠聰明，而是因為沒有限制。

這一層會定義：

- 哪些事必須做
- 哪些事不能做
- 什麼情況一定要追問
- 什麼情況不能亂下判斷
- 什麼情況要保守
- 什麼情況要請人類決定

### 6. 輸出格式

最後才是它怎麼表達。

例如：

- 要不要先給結論
- 要不要分點
- 要不要列風險
- 要不要有建議等級
- 要不要提供下一步

也就是說，我做的不是單純「調語氣」。

我是把客戶需要的專家行為，拆成一套 AI 可以複製的結構。

## 這個 repo 怎麼證明我做得到

這個 repo 不是概念簡報，而是實際做出來的研究與抽取成果。

我做了以下事情：

1. 把兩個成熟的 skill repo 完整拉下來
2. 逐份掃描它們的 `SKILL.md`、`CLAUDE.md`、`AGENTS.md`、`README.md`
3. 為每一份文件建立 extraction card
4. 把每個 skill 的核心拆成標準欄位：
   - 一句話定位
   - 核心人格特質
   - 核心思考框架
   - 核心行為規則
   - 提問方式
   - 審查維度
   - 輸出格式要求
   - 適用場景
   - 原文精華摘錄
   - 和其他 skill 的關聯
5. 再把全部資料做成：
   - 總索引
   - 分類彙整
   - 分析報告
   - 一份整合用的 persona 草案

這代表我不是在「猜」怎麼做 agent。

我是用可拆解、可比對、可重組的方法，在設計 agent。

## 這個專案實際解決了哪些問題

### 1. 解決 AI 太像通用助手的問題

很多 AI 什麼都能答，但不像任何一種真實專家。

我做的事情，是把專家的做事方式拆出來，讓 AI 可以穩定扮演那個角色。

### 2. 解決 AI 沒有人格、沒有一致性的問題

如果沒有固定人格，AI 每次回話都像不同人。

這對品牌、客服、顧問型服務、內部流程都很傷。

### 3. 解決文件很多但 AI 不會照做的問題

很多公司不是沒有資料，而是資料沒有變成 AI 可以執行的 skill。

文件只是資料，skill 才是行為。

### 4. 解決 AI 不知道什麼時候該追問、什麼時候該保守的問題

很多 AI 看起來很聰明，但其實沒有明確決策邏輯。

所以該追問的時候不追問，不該亂判斷的時候亂判斷。

### 5. 解決客製化 agent 難以規劃的問題

很多人知道自己想要什麼 AI，但不知道怎麼把需求變成結構。

我做的事情，就是把需求拆成角色、人格、流程、規則與輸出，讓客製化變成可操作的工程。

### 6. 解決 agent 後續越做越亂的問題

如果一開始沒有把 skill 基礎整理清楚，後面每加一個角色、每多一個場景，整套系統都會變亂。

這個 repo 展示的做法，是先拆、先整理、再合成。

## 技術版說明

如果你想看這個專案的工程結構、資料格式與輸出內容，下面是技術版摘要。

### Repo 結構

```text
.
├── README.md
├── generate_skill_extraction.py
├── skill-extraction-spec.md
├── skill-extraction-analysis.md
├── skill-persona.md
└── skill-extraction/
    ├── repos/
    │   ├── gstack/          # 上游來源，submodule
    │   └── superpowers/     # 上游來源，submodule
    ├── raw-cards/
    │   ├── gstack/
    │   └── superpowers/
    ├── 00_總索引.md
    ├── 01_分類彙整.md
    └── raw-cards.zip
```

### 核心檔案

- `generate_skill_extraction.py`
  - 產生器腳本，負責掃描來源 repo、整理內容、輸出卡片與索引
- `skill-extraction-spec.md`
  - 本次抽取工作的規格與欄位定義
- `skill-extraction-analysis.md`
  - 對 raw data 的二次分析，說明哪些技能最有整合價值
- `skill-persona.md`
  - 基於抽取與分析後形成的整合式 persona / framework 草案
- `skill-extraction/raw-cards/`
  - 研究用核心資料集，每張卡對應一份來源文件
- `skill-extraction/00_總索引.md`
  - 全部卡片的統計、清單與跨 repo 關聯矩陣
- `skill-extraction/01_分類彙整.md`
  - 依分類標記重新整理的總表
